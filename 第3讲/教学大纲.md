# 第3讲 教学大纲：Spec-driven 协作建模与 Python 工程化

## 一、本讲教学目标

### 知识目标

1. 掌握 Python 包结构（`__init__.py`、模块划分、绝对/相对导入）
2. 掌握类型注解（typing 模块常用类型、函数签名标注）
3. 掌握异常处理（try/except 层级、自定义异常、上下文管理器）
4. 掌握 pytest 测试框架（断言、fixture、参数化）
5. 理解 Spec-driven 开发范式（Constitution/Context/Tasks 三文档、Speckit 五步工作流）

### 能力目标

1. 能将一段散装分析脚本改造为模块化 Python 包
2. 能为 AI 生成的代码编写测试用例并验证正确性
3. 能编写 speckit.md 规约文档约束 AI 的实现行为
4. 能运用 TDD + AI 的方式开发新功能（先测试→AI实现→验收）

### 态度目标

1. 建立"代码质量是工程底线"的意识，区分"能跑"和"可靠"
2. 认识到 AI 越强大、约束越重要——不约束的 AI 像没有图纸的施工队
3. 形成"先设计后实现"的工程化思维

## 三、两条主线融合设计

### 知识主线（Python 工程化）
```
模块化组织 → 类型注解 → 异常处理 → 测试 → 代码质量 → Spec-driven
```

### 工具主线（Vibe Coding — AI 作为受控开发者）
```
AI辅助重构 → TDD+AI → Spec约束 → 验收循环（Spec→AI→pytest→人）
```

### 融合节点

| 知识点 | AI 角色 | 协作方式 |
|--------|---------|---------|
| 模块拆分 | 重构助手 | 描述目标结构 → AI 拆分代码 → 检查依赖关系 |
| 类型注解 | 标注助手 | AI 自动推断类型 → 人审查 Optional/Union 是否合理 |
| 异常处理 | 实现者 | 指定异常层级 → AI 补全 try/except → 审查覆盖面 |
| 测试 | 被测对象 | 人写测试用例 → AI 实现功能 → pytest 裁判 |
| Spec-driven | 受控开发者 | speckit.md 定义边界 → AI 在约束内实现 → 验收 |

**本讲 AI 角色定位 = 受控开发者**：AI 不再自由发挥，而是在你定义的 Spec 约束内实现。你的角色从"审查者"升级为"架构师"。

## 四、重点难点与应对策略

### 重点

| 重点内容 | 教学策略 |
|----------|----------|
| 模块化拆分原则 | 用贯穿案例实演：100行脚本→4模块，对比前后 |
| TDD + AI 工作流 | 现场演示完整循环：写测试→AI实现→失败→纠正→通过 |
| speckit.md 编写 | 给模板 + 反例对比（无Spec的AI输出 vs 有Spec的AI输出） |

### 难点

| 难点内容 | 应对策略 |
|----------|----------|
| 模块间依赖和循环导入 | 画依赖关系图，演示循环导入报错及解决方案 |
| 什么时候用 Spec 什么时候用 Prompt | 给明确决策树：单函数→Prompt；多模块→Spec |
| 测试用例设计（正常/边界/异常） | 提供 checklist：正常输入、空输入、类型错误、边界值 |
| Constitution 粒度把控 | 过细→限制AI创造力；过粗→没约束效果。给出"刚好"示例 |

## 五、课前准备与课后作业

### 课前准备

1. **环境确认**：CloudStudio 中已安装 pytest、ruff（`pip install pytest ruff`）
2. **代码准备**：将实验1的数据清洗代码保存为 `analysis_script.py`（作为改造起点）

   - 了解 Python 包和模块的基本概念（import 语句）
   - 阅读 pytest 官方 Getting Started（前半部分）
4. **思考**：你在第1-2讲的实验中遇到过哪些"代码能跑但不够好"的问题？

### 课后作业

1. **实验完成**：实验三全部 4 个任务（模块拆分/类型注解/测试/Spec-driven）
2. **验证通过**：`pytest` 全绿 + `ruff check .` 无报错
3. **提交材料**：项目目录 + speckit.md + 测试截图 + AI 对话截图
