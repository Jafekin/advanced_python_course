### Task 2：RAG 知识检索工具（20 分）

**技术要求**：

#### 2.1 文档处理流水线（8 分）

实现从原始文档到可检索向量库的完整链路：

```
原始文档（PDF / TXT / MD）
    ↓
文本提取（markitdown / PyMuPDF / pdfplumber）
    ↓
分块策略（Chunk）
    ↓
向量化（Embedding）
    ↓
持久化向量库（ChromaDB）
```

**分块策略须说明并实验对比**（这是本 Task 的核心技术点）：

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| 固定大小切分 | 按 token 数硬切，重叠 `overlap` 个 token | 结构均匀的文档 |
| 递归字符切分 | 按段落 → 句子 → 字符层级递归 | 通用文本（推荐起点） |
| 语义切分 | 计算相邻句子嵌入相似度，相似度骤降处切断 | 内容密度不均匀的长文 |

你需要选择其中**至少两种策略**，用相同查询对比检索质量，给出结论。

#### 3.2 工具描述 Prompt（3 分）

LangChain 中每个工具的 `description` 字段是 LLM 决定"是否调用这个工具"的唯一依据。设计高质量的工具描述：


### Task 4：双模式 Agent 引擎（40 分）

这是本大作业的核心，也是技术难度最高的部分。

**目标**：实现一个支持 **ReAct** 和 **Plan-and-Execute** 两种推理模式的 Agent 引擎，两种模式可由用户在运行时切换，并且系统会在运行中**完整输出推理过程**（Thinking + 工具调用链）。

#### 4.1 ReAct 模式（12 分）

ReAct（Reason + Act）：每一步先思考，再行动，看到结果，再思考下一步。适合**短任务、动态问题**。

```
用户问题
   ↓
Thought: 我需要先检索文档了解背景...
Action: rag_tool
Action Input: "..."
Observation: [检索结果]
   ↓
Thought: 现在我有了背景，需要进一步...
Action: [下一个工具]
...
   ↓
Final Answer: [最终回答]
```

**实现要求**：

```python
from langchain.agents import create_react_agent, AgentExecutor
from langchain import hub

# 使用 ReAct Prompt 模板
react_prompt = hub.pull("hwchase17/react")

react_agent = create_react_agent(llm=llm, tools=tools, prompt=react_prompt)

react_executor = AgentExecutor(
    agent=react_agent,
    tools=tools,
    verbose=True,          # 输出推理过程
    return_intermediate_steps=True,  # 返回中间步骤
    max_iterations=8,
    handle_parsing_errors=True,
)
```

**推理过程可视化要求**：每一步的 Thought / Action / Observation 必须在界面中实时显示（不能只显示最终答案）。

#### 4.2 Plan-and-Execute 模式（15 分）

Plan-and-Execute：先生成完整计划，再逐步执行，适合**复杂多步骤任务**（如"帮我分析这份文档，建模，然后出报告"）。

```
用户任务（复杂）
   ↓
═══ 规划阶段（Planner LLM）═══
Plan:
  Step 1: 检索文档中的相关数据
  Step 2: 对检索到的数据进行可视化
  Step 3: 运行 K-means 聚类分析
  Step 4: 综合以上结果生成报告
   ↓
═══ 执行阶段（Executor LLM）═══
[执行 Step 1] → 调用 rag_tool → 得到结果
[执行 Step 2] → 调用 visualization_tool → 得到图表
[执行 Step 3] → 调用 ml_tool → 得到聚类结果
[执行 Step 4] → 综合生成最终报告
```

**实现要求**：

```python
from langchain_experimental.plan_and_execute import (
    PlanAndExecute,
    load_agent_executor,
    load_chat_planner,
)

planner = load_chat_planner(llm)
executor = load_agent_executor(llm, tools, verbose=True)

plan_execute_agent = PlanAndExecute(
    planner=planner,
    executor=executor,
    verbose=True,
)
```

**规划结果可视化要求**：执行前须在界面中显示完整 Plan（步骤列表），每步执行完毕后标记完成状态。

#### 4.3 模式切换与融合（8 分）

实现运行时模式切换，并探索两种模式的融合策略：

**切换实现**（Gradio 界面中加一个单选按钮）：

```python
def run_agent(user_input: str, mode: str) -> str:
    if mode == "ReAct":
        result = react_executor.invoke({"input": user_input})
    elif mode == "Plan-and-Execute":
        result = plan_execute_agent.invoke({"input": user_input})
    elif mode == "Auto":
        # 自动判断：简单问题用 ReAct，复杂任务用 P&E
        result = auto_dispatch(user_input, llm, react_executor, plan_execute_agent)
    return format_result(result)
```

**Auto 模式（融合策略）**：实现一个简单的任务复杂度判断器，根据问题长度、是否包含"步骤/分析/报告/对比"等关键词，自动选择推理模式。在报告中展示判断逻辑和测试结果。

**评分细则**：
- 切换功能正常运行：4 分
- Auto 模式有合理的判断逻辑：4 分

#### 4.4 推理过程可视化（5 分）

这是本大作业最能体现"理解深度"的部分。要求在 Gradio 界面中实时展示：

## 七、常见问题（FAQ）

**Q：Agent 主题必须和课程案例一样是"航天"或"数据分析"吗？**  
A：完全不需要。你可以用任何主题——个人知识库、编程助手、旅行规划、食谱推荐……只要技术骨架（Task 1-4）完整实现即可。选一个你真正感兴趣的场景，报告会写得更有说服力。

**Q：Ollama 模型太大，我的电脑跑不动怎么办？**  
A：推荐 4bit 量化版本：`qwen2.5:7b-instruct-q4_K_M`（约 4.5GB，8GB 内存可运行）。如果实在无法本地运行，可以用 云端 的 API，但需要在报告中说明原因（说明选型分析）。

**Q：Plan-and-Execute 的 LangChain 接口不稳定怎么办？**  
A：`langchain_experimental` 包的 Plan-and-Execute 接口确实有版本差异。如果遇到问题，可以自行实现一个简化版：用一个 LLM 调用生成计划（输出有序步骤列表），再用 ReAct Agent 逐步执行每个步骤。在报告中说明你的实现方案，教师会根据实际效果评分。

**Q：RAG 的 Embedding 模型需要联网下载，实验室没有外网怎么办？**  
A：`all-MiniLM-L6-v2` 可在 Hugging Face 镜像下载。或者使用 Ollama 的 Embedding 接口（`ollama pull nomic-embed-text`），完全本地运行。在 Notebook 顶部说明你的替代方案即可。

**Q：MCP 工具和 Skill 函数有什么区别，必须用哪种？**  
A：两者都可以作为 Agent 工具注册进 LangChain。Skill 函数就是普通 Python 函数（更简单，第6讲已学）；MCP 工具是通过 MCP 协议暴露的外部服务（更标准化，是扩展 D 的内容）。必做部分用 Python 函数即可；想冲扩展加分才需要接入 MCP。

**Q：扩展功能可以自己想一个没有列出来的吗？**  
A：可以，这正是扩展 I 的设计意图。唯一要求是：在报告中清晰说明这个功能的技术实现方式和它给 Agent 带来了什么价值，由教师评定加分。
