## 自学导航卡

- **前置依赖：** 已掌握 Rules 基础，能引用文件与目录

- **卡点入口：** 若输出混乱，立刻缩减上下文并重开单任务对话

## 一、为什么上下文如此重要？

AI 不是全知全能的。它的每一次回答都 **完全基于你给它的信息**。

```
信息不足 → AI 瞎猜 → 生成的代码和你的项目格格不入

信息过多 → AI 混乱 → 抓不住重点，输出质量下降

信息精准 → AI 高效 → 生成的代码直接可用
```

## 一、什么是上下文污染？

当你给 AI 的信息中包含了 **不相关** 或 **误导性** 的内容时，AI 的输出质量会急剧下降。

**常见污染场景：**

| 污染类型 | 描述 | 后果 |

|---------|------|------|

| **陈旧信息** | 引用了已经废弃的旧代码 | AI 生成兼容旧代码的方案 |

| **噪声过多** | 把整个 500 行文件都扔给 AI | AI 抓不住重点 |

| **矛盾信息** | 对话前面说"用 REST"，后面又说"用 GraphQL" | AI 左右为难 |

| **跨任务污染** | 一个对话窗口讨论了 3 个不同的功能 | AI 把功能搞混 |

## 二、最小上下文原则（Minimum Context Principle）

> **只给 AI 看它完成当前任务所需的最少信息。**

### 实操对比

**不推荐示范：上下文过载**

## 二、Step 1：准备 API 文档

首先创建 API 文档文件 `docs/api-spec.md`：

<details style="margin-bottom: 8px;">

<summary style="color: #f57c00; font-weight: 500; cursor: pointer;">详细文件内容</summary>

```markdown
# 用户管理 API



## 获取用户列表

- GET /api/users

- Query: page(number), pageSize(number), keyword(string)

- Response: { code: 200, data: { list: User[], total: number }, message: "success" }



## 获取用户详情

- GET /api/users/:id

- Response: { code: 200, data: User, message: "success" }



## 创建用户

- POST /api/users

- Body: { name: string, email: string, role: "admin" | "user" }

- Response: { code: 200, data: User, message: "success" }



## 更新用户

- PUT /api/users/:id

- Body: Partial<User>

- Response: { code: 200, data: User, message: "success" }



## 删除用户

- DELETE /api/users/:id

- Response: { code: 200, data: null, message: "success" }



## User 类型

- id: number

- name: string

- email: string

- role: "admin" | "user"

- avatar: string

- createdAt: string (ISO 8601)
```

</details>

## 三、Step 2：让 AI 阅读文档并生成类型

```
阅读 @Files docs/api-spec.md，然后帮我：

1. 在 src/types/user.ts 中定义完整的 TypeScript 类型

2. 包含 API 请求参数和响应的类型

3. 包含通用的 API 响应包装类型
```

## 四、Step 3：让 AI 生成 API 请求层

```
基于 @Files src/types/user.ts 中的类型定义，

在 src/lib/api/user.ts 中创建完整的 API 请求函数：

1. 使用 fetch 封装

2. 统一错误处理

3. 支持 TypeScript 类型推导

4. baseURL 从环境变量读取
```

## 五、Step 4：让 AI 生成自定义 Hook

```
基于 @Files src/lib/api/user.ts 中的 API 函数，

创建以下 Hook：

1. useUsers(params) —— 获取用户列表，支持分页和搜索

2. useUser(id) —— 获取单个用户详情

3. useCreateUser() —— 创建用户（包含 loading 和 error 状态）

4. useUpdateUser() —— 更新用户

5. useDeleteUser() —— 删除用户（带确认）



使用 useSWR 实现数据获取和缓存。
```

## 六、Step 5：让 AI 生成页面组件

```
基于 @Files src/hooks/useUsers.ts 中的 Hook，

创建一个用户管理页面，包含：

1. 顶部搜索栏 + "新增用户"按钮

2. 用户列表表格（姓名、邮箱、角色、创建时间、操作）

3. 分页控件

4. 新增/编辑用户的弹窗表单

5. 删除确认弹窗



使用 TailwindCSS 样式，参考 Linear 的设计风格。
```

## 八、关键总结

注意每一步我们是如何 **分层构建上下文** 的：

```
API 文档（源头）

↓ 引用文档

类型定义（数据结构）

↓ 引用类型文件

API 请求函数（网络层）

↓ 引用 API 文件

自定义 Hook（状态层）

↓ 引用 Hook 文件

页面组件（视图层）
```

每一步只引用 **上一步的产出**，而不是把所有东西一股脑扔给 AI。这就是 **分层上下文投喂** 的精髓。

## 一、上下文"预热"技巧

在正式提需求之前，先让 AI "认识"你的项目。

**适用场景**

- 你刚切到一个新项目

- 你要连续做 2-3 个相关功能

- 你希望 AI 输出风格与现有代码保持一致

**错误用法 vs 正确用法**

<div style="padding: 10px; background: #ffebee; border-left: 4px solid #f44336; border-radius: 4px; margin: 10px 0;">

<strong>错误用法（直接开干）：</strong>

```markdown
帮我做一个用户列表页面。
```

</div>

<div style="padding: 10px; background: #e8f5e9; border-left: 4px solid #4caf50; border-radius: 4px; margin: 10px 0;">

<strong>正确用法（先预热再提需求）：</strong>

```markdown
我先让你了解一下项目结构：



@Folders src



目前项目使用 Next.js 14 + TailwindCSS，

状态管理用 Zustand，数据获取用 useSWR。



接下来我会让你帮我实现几个功能，

请在后续的回答中保持与项目现有模式的一致性。
```

</div>

**实际案例**

- 需求：新增“用户搜索 + 分页”

- 未预热结果：AI 可能改用 React Query、改造目录结构

- 预热后结果：更容易沿用 `useSWR + Zustand` 现有模式，返工更少

## 二、上下文"锚点"技巧

当对话变长时，用"锚点"帮 AI 回忆关键信息。

**适用场景**

- 会话已经超过 10 轮

- 中间切过多个任务（列表、详情、导出）

- 你发现 AI 开始“忘记”既定类型或文件位置

**错误用法 vs 正确用法**

<div style="padding: 10px; background: #ffebee; border-left: 4px solid #f44336; border-radius: 4px; margin: 10px 0;">

<strong>错误用法（不回顾上下文）：</strong>

```markdown
现在加导出 CSV。
```

</div>

<div style="padding: 10px; background: #e8f5e9; border-left: 4px solid #4caf50; border-radius: 4px; margin: 10px 0;">

<strong>正确用法（先锚点再继续）：</strong>

```markdown
（接上面的用户管理功能）



回顾一下：

- 我们用的是 @Files src/types/user.ts 中的 User 类型

- API 层在 @Files src/lib/api/user.ts



现在帮我给用户列表添加"导出 CSV"功能。
```

</div>

**实际案例**

- 症状：AI 把导出字段写成 `user_name`，而现有类型是 `name`

- 加锚点后：AI 会优先复用 `User` 类型，字段漂移明显减少

## 三、负面上下文（告诉 AI "不要什么"）

有时候告诉 AI "不要做什么" 比 "要做什么" 更有效。

**适用场景**

- 你只想局部重构，不想改外部行为

- 项目已上线，接口与样式不能被破坏

- 团队规范明确限制依赖与命名

**错误用法 vs 正确用法**

<div style="padding: 10px; background: #ffebee; border-left: 4px solid #f44336; border-radius: 4px; margin: 10px 0;">

<strong>错误用法（目标宽泛，边界缺失）：</strong>

```markdown
帮我优化这个组件。
```

</div>

<div style="padding: 10px; background: #e8f5e9; border-left: 4px solid #4caf50; border-radius: 4px; margin: 10px 0;">

<strong>正确用法（明确禁止项）：</strong>

```markdown
帮我重构这个组件：

@Files src/components/Dashboard.tsx



要求：

- 不要改变任何对外的 Props 接口

- 不要引入新的第三方库

- 不要修改现有的 CSS 类名

- 只拆分内部逻辑，提取子组件
```

</div>

**实际案例**

- 未给禁止项：AI 可能顺手改 Props、替换样式体系

- 给出禁止项后：输出会聚焦在“内部拆分与可读性优化”，可直接合并概率更高

## 练习 3：整理你的上下文策略（必做）

为你最常做的 3 种开发任务，各设计一套上下文投喂策略：

- 需要引用哪些文件？

- 按什么顺序提供？

- 哪些信息需要排除？

本讲结束时，你应该能做到：

- 熟练使用 @Files / @Folders / @Terminal / @Docs

- 理解并能应用“最小上下文原则”

- 能按“分层投喂”的顺序组织信息，提高稳定性

- 能识别常见的上下文污染并做排查

- 用一套清晰的上下文包完成一次 API 对接练习
