在使用之前，我们需要了解一下这个 `SKILL.md` 文档的最基本的格式。

- 一个目录（技能名）

- 里面一个必须存在且大小写敏感的文件：`SKILL.md`

`SKILL.md` 的开头需要 YAML 元数据（至少 `name` 和 `description`）：

```markdown
---

name: 你的技能名称

description: 简要描述此技能的功能以及何时使用它

---



# 你的技能名称



## 使用说明

...
```

- **`description` 写得越具体，skill 越容易被触发**。

## Skills 如何运作

根据需要分阶段加载信息，而不是预先获取上下文。

技能可以包含三种类型的内容，每种内容的加载时间各不相同：

### 级别 1:元数据（始终加载）

该技能的 YAML 前置元数据提供发现信息：

```markdown
---

name: 你的技能名称

description: 简要描述此技能的功能以及何时使用它

---
```

会在启动时加载这些元数据并将其包含在系统提示符中。这种轻量级方法意味着您可以安装许多技能而不会受到上下文影响；并且只知道每个技能的存在以及何时使用它。

### 级别 2:指令（触发时加载）

SKILL.md 的主体内容包含程序性知识：工作流程、最佳实践和指导原则：

```markdown
# 你的技能名称



## 使用说明

...
```

当您请求的内容与技能描述相符时，会通过 bash 从文件系统中读取 SKILL.md 文件。只有这样，该文件的内容才会显示在上下文窗口中。

### 级别 3:资源和代码（按需加载）

技能可以捆绑其他材料：

```text
.codebuddy/skills/

meeting-minutes/（负责把会议对话变成结构化纪要）

SKILL.md

expense-reimbursement/（负责把“报销标准/流程”固化成可复用的规则与输出）

SKILL.md

reference.md
```

包含专门指导和工作流程的附加 Markdown 文件（reference.md）

### 如何运作总结

仅在被引用时才会访问这些文件。文件系统模型意味着每种内容类型都有其不同的优势：指令提供灵活的指导，代码提供可靠性，资源用于事实查找。

等级|加载时|内容

---|---|---

一级：元数据|始终（在启动时）|来自 YAML 前置元数据的 name 和 description

二级：说明|技能触发时|SKILL.md 正文包含说明和指导

三级及以上：资源|根据需要|通过 bash 执行的捆绑文件未将内容加载到上下文中

# Meeting Minutes（项目会议纪要）



## Input（缺一项先问）

- 会议文本：聊天记录/录音转写/要点

- 背景与目标（Background & Goals）

- 结论摘要（Executive Summary，≤10 行）

- 决策（Decisions：事项 | 结论 | 影响范围）

- 待办（Action Items：事项 | Owner | Deadline | 风险/依赖）

- 未决问题（Open Questions）

- 费用/报销（若涉及）：费用项 | 是否可报 | 标准/上限 | 需要材料 | 下一步

1. 判断信息是否足够；不足先问 3 个最关键问题（别猜）

3. 检测是否出现费用/报销/预算相关内容

4. 若出现：调用 expense-reimbursement，并把结果写入“费用/报销”小节



## Acceptance

- 纪要可以直接进项目文档库（不用二次整理）

#### `reference.md` 放什么？

你不需要写成制度全文，但至少要覆盖：

- **费用类型 → 标准/上限/不可报情况**

- **金额阈值 → 审批链分支**

- **材料清单（按类型）**

- **例外与常见驳回原因**（这部分最能减少来回追问）

如果担心敏感信息，你可以把“金额阈值”写成区间占位符（例如 `T1/T2/T3`），但要保证逻辑分支完整。

## 它们分别解决什么问题

- **Agent Skills**

- 把你的经验/方法论写成可复用的“标准作业流程（SOP）”

- 目标是：**输出稳定、结构一致、可验收、可复现**

- 典型例子：会议纪要固定输出“背景-决策-待办-风险”；报销固定输出“材料清单-审批步骤-缺失信息列表”

- **MCP（Model Context Protocol）**

- 是一套“让模型能安全、标准化调用外部能力”的协议/接口

- 目标是：**让 AI 能连上工具和数据源**（查库、查文档、拉工单、搜知识库、调用内部服务等）

- 典型例子：让 AI 去读取某个知识库、查询报销系统、拉取项目数据、调用测试/部署工具

<img src='img/Clipboard_Screenshot_1769499804.png' width=800px />

## 核心差异（对照表）

| 维度 | Agent Skills | MCP |

|---|---|---|

| 本质 | 工作流/规则/模板（行为规范） | 工具与数据接入协议（能力接口） |

| 重点 | 让输出“按你想要的方式”稳定产出 | 让模型“拿到数据、用上工具” |

| 没有它会怎样 | AI 每次写法不一、口径飘、难复用 | AI 只能凭已有上下文“猜”，无法实时查/调 |

| 典型产物 | `SKILL.md` 里的触发条件、约束、步骤、输出格式 | 一组可调用的 tools（查询、检索、执行动作等） |

## 什么时候用哪个？

- 你想要“**每次都按这个格式写、按这个流程做**” → 用 **Skills**

- 你需要“**去某个系统/知识库拿数据、调用某个工具**” → 用 **MCP**

- 最强组合：**Skills 负责指挥流程，MCP 负责提供手和眼睛**

- 例如“报销技能”规定：先问清字段 → 再用 MCP 去查制度/单据状态 → 最后按模板输出可执行清单。
