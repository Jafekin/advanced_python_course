## 一、实验目标

### 知识目标
1. 掌握大模型本地部署：用 Ollama 起 OpenAI 兼容 API 服务（或用云端 API 替代）
2. 掌握 RAG 完整流水线：文档加载 → 文本分块 → 向量化 → ChromaDB 存储 → 语义检索 → Prompt 拼接 → LLM 生成

2. 用 `@rag_speckit.md` 引用驱动 CodeBuddy 按规约生成 RAG 代码
3. 用 Inline Edit 调整 `chunk_size`、`top_k` 等参数，对比英文分块与中文分块的效果差异
4. 用 `@综述论文.pdf` 引用让 AI 了解文档结构，辅助写英文分块函数和 Prompt 模板
5. 形成"知识库离线建一次、查询在线每次跑"的 RAG 工程思维

## 三、Spec-driven 核心概念

### 什么是 Speckit？

Speckit 是 GitHub 开源的 Spec-driven 开发工具，核心理念是**先写规约，再让 AI 按规约执行**。

### rag_speckit.md 的三段式结构

```markdown
# 学术论文 RAG 问答系统规约

## constitution（不可违反的硬约束）
- 嵌入模型：sentence-transformers/all-MiniLM-L6-v2（384维，英文为主）
  - 进阶对比：BAAI/bge-m3（1024维，多语言，用于跨语言检索对比）
- 向量库：ChromaDB，持久化到 ./chroma_db
- 分块策略：英文递归字符分块，chunk_size=600，overlap=80
  - 分隔符优先级：". \n" → "\n\n" → "\n" → ". " → "; " → ", "
- 检索：top_k=5，余弦相似度
- LLM：本地 Ollama 或云端 OpenAI 兼容 API
- 防幻觉：Prompt 必须明确"资料中未提及就说不知道"
- 引用标注：回答必须用 [1][2] 标注来源 chunk

## context（数据背景 + 分析目标）
数据：综述论文.pdf（ACM TOSEM 2025，"LLM for Mobile: An Initial Roadmap"，29页英文综述）
内容：9 章——Introduction / Dataset / App Development / Serving / Security / Framework APIs / Runtime Monitoring / Discussion / Related Work
目标：搭建可交互的学术论文问答系统，支持中英文混合查询

## 任务列表（可执行、可验证的步骤）
- [ ] 任务1：加载英文 PDF 并实现英文递归分块
- [ ] 任务2：向量化并存入 ChromaDB（metadata 含章节字段）
- [ ] 任务3：实现检索 + 生成（支持中英文查询）
- [ ] 任务4：实现 MQE 或 HyDE 高级检索并对比 Recall@5
```

### 使用方式

每次执行任务时，在 Prompt 中附带 `@rag_speckit.md` 引用：

```
@rag_speckit.md @综述论文.pdf
执行任务1：加载综述论文.pdf 并实现英文递归字符分块（分隔符优先级见 speckit）
```

CodeBuddy 会自动读取规约，按 constitution 约束生成代码。

## 五、实验任务

### 任务1：英文文档加载与分块（25 分）

**目标**：把英文 PDF 转成 Markdown，实现**英文递归字符分块**函数（与 notebook 的中文分块不同）。

**要求**：
1. 用 `markitdown` 把 `综述论文.pdf` 转成 Markdown 文本（注意：英文 PDF 转换后含大量换行和页眉页脚，需清洗）
2. 实现英文递归字符分块函数 `chunk_recursive_en(text, chunk_size=600, overlap=80)`，分隔符优先级：
   ```
   ". \n"  →  "\n\n"  →  "\n"  →  ". "  →  "; "  →  ", "  →  " "
   ```
   （注意：英文句末是 `. ` 而非中文的 `。`；先按段落 `\n\n` 切，再按句号 `. ` 切）
3. 对比三种 `chunk_size`（400 / 600 / 1000）的分块效果，记录块数、平均块长、是否有跨段切断
4. 写 `rag_speckit.md` 的 constitution 和 context 部分

**Vibe Coding 提示**：

```

1. 用 markitdown 把英文 PDF 转 Markdown（注意清洗页眉页脚 "ACM Transactions..."、"LLM for Mobile..."）
2. 实现英文递归字符分块函数（分隔符优先级见 speckit）
3. 对比 chunk_size=400/600/1000 的分块效果，打印块数和平均块长
```

**交付物**：

### 任务4：高级检索对比（20 分）

**目标**：实现 MQE 或 HyDE 高级检索，对比基础检索的效果提升（跨语言场景下效果差异更明显）。

**要求**（二选一）：

**选项 A：多查询扩展 MQE**
1. 实现 `expand_query_mqe(query, n=3)`：用 LLM 生成 3 个扩展查询
   - **注意**：扩展查询要与原查询**同语言**（中文问题扩展成中文查询，英文问题扩展成英文查询）
2. 实现 `retrieve_mqe(query, top_k=5)`：并行检索 + 合并去重
3. 对比基础检索 vs MQE 在 5 个预设问题上的 Recall@5

**选项 B：假设文档嵌入 HyDE**
1. 实现 `hyde_retrieve(query, top_k=5)`：先让 LLM 生成假设答案，再用假设答案检索
   - **注意**：假设答案要用**与文档一致的语言（英文）**生成，这样向量才与英文文档对齐
   - 提示：中文问题先用 LLM 翻译成英文，再让 LLM 生成英文假设答案，再用假设答案检索英文文档
2. 对比基础检索 vs HyDE 在 5 个预设问题上的 Recall@5

## 八、常见问题

**Q1：CloudStudio 没装 Ollama 怎么办？**
A：用云端 OpenAI 兼容 API 替代。修改 `client = OpenAI(base_url="云端地址", api_key="你的key")`，`MODEL` 改成对应模型名。RAG 流水线代码完全不用改。

**Q2：sentence-transformers 模型下载慢？**
A：设置镜像源：
```python
import os
os.environ["HF_ENDPOINT"] = "https://hf-mirror.com"
# 然后再加载模型
```

**Q3：ChromaDB 报错"collection already exists"？**
A：先删除已有 collection：
```python
try:
    chroma_client.delete_collection("llm_mobile_survey")
except:
    pass
```

**Q4：英文 PDF 转出来有很多页眉页脚怎么办？**
A：清洗步骤：
```python
# 过滤掉页眉页脚（ACM 期刊常见）
lines = text.split("\n")
cleaned = [ln for ln in lines if not any(
    skip in ln for skip in
    ["ACM Transactions", "LLM for Mobile: An Initial Roadmap", "D. Chen et al."]
)]
text = "\n".join(cleaned)
```

**Q5：中文查询检索英文文档效果差？**
A：这是跨语言检索的难点，可选三种方案：
1. 换多语言嵌入模型 `BAAI/bge-m3`（推荐，加分项）
2. 用 MQE 把中文查询扩展成中英文混合查询
3. 用 HyDE 让 LLM 先生成英文假设答案再检索

**Q6：LLM 回答有幻觉？**
A：加强 Prompt 约束：
- 明确"只根据参考资料回答"
- 明确"资料中未提及就说'资料中未提及'"
- 降低 temperature 到 0.1
- 要求带引用标注 `[1][2]`
